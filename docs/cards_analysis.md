# Аналіз карт гри (Hero Cards) — повний каталог для імплементації

Документ описує структуру та всі типи даних з `cards.json` (72 героїв) для подальшої імплементації на бекенді та фронтенді.

---

## 1. Загальна структура картки

Кожен елемент масиву — об'єкт з полями:

| Поле      | Тип              | Опис |
|----------|------------------|------|
| `name`   | `string`         | Назва карти (англійською). |
| `fraction` | `string`       | Фракція: `"Highlanders"` \| `"Imperials"` \| `"Undead"` \| `"Waterfolk"`. |
| `markers` | `MarkersObject`  | Вплив на трек червоної/зеленої фішки (див. розділ 2). |
| `ability` | `AbilityObject \| null` | Здібність карти або її відсутність. |

**Важливо:** у поточному бекенді використовується поле `faction` (не `fraction`). При імпорті потрібно узгодити назву поля або мапити `fraction` → `faction`.

---

## 2. Маркери (markers) — вплив на трек сили

### 2.1 Структура `markers`

```ts
interface MarkersObject {
  red: number | "X" | "-X";   // зміна червоного маркера (Імперія)
  green: number | "X" | "-X";  // зміна зеленого маркера (Племена)
  logic: MarkerLogic;          // як застосовувати red/green (див. нижче)
  red_alt?: number;           // опційно: альтернатива для red при OR-логіці
  green_alt?: number;         // опційно: альтернатива для green при OR-логіці
}
```

- **Числові значення:** від `-3` до `2` (діапазон у файлі).
- **Спеціальні:** `"X"` — змінна (обчислюється згідно з ability/умовами), `"-X"` — мінус змінна.

### 2.2 Типи логіки маркерів (`logic`)

| Значення | Опис (для бекенду/фронту) |
|----------|----------------------------|
| `AND` | Обидва маркери застосовуються: +red і +green за значеннями з полів. |
| `OR` | Гравець обирає один із двох варіантів (наприклад, red або green). Часто разом з `red_alt`/`green_alt`. |
| `OR_NEG` | Варіант OR з від’ємним другим варіантом (наприклад, +2 green або -2 green). |
| `OR_NEG_DECIDE_LEFT` | OR з від’ємним варіантом і рішенням «лівий/правий» (наприклад, +2 / -2 з вибором сторони). |
| `AND_OR` | Комбінована логіка: обидва маркери, але з можливістю вибору варіанту (наприклад, 2r+2g як AND або OR за правилами гри). |
| `LEADING_MARKER` | Вплив залежить від того, який маркер «веде» (leading). Не фіксовані red/green, а динаміка за позицією на треку. |

**Приклади з файлу:**
- `OR` + `green_alt: 2` — вибір між (red, green) і (red, 2) залежно від умов.
- `OR` + `red_alt: 2` — аналогічно для червоного.
- `LEADING_MARKER` — використовується разом з ability `Move_Markers` (рух маркерів за «leading»/«behind»).

## 3. Здібності (ability)

### 3.1 Загальні поля ability

- **`action`** (обов’язково) — тип дії (див. таблицю нижче).
- Додаткові поля залежать від `action`: `target_player`, `target_zone`, `source`, `visibility`, `filters`, `count`, `distribution`, `condition`, `x_source`, `targets`, `options`, `effects`, `logic` тощо.

### 3.2 Список усіх типів дій (`action`)

| Action | Опис (коротко) | Типові додаткові поля |
|--------|----------------|------------------------|
| **Kill** | Вбити ціль(і) у партії. | `target_player?`, `target_zone`, `visibility`, `filters?` |
| **Bury** | Поховати карт(и) (наприклад, в цвинтар). | `target_player`, `target_zone`, `visibility` |
| **Flip** | Перевернути карт(и) (face_down ↔ face_up). | `target_player`, `target_zone` |
| **Swap** | Обміняти карту/карти між зонами. | `source`, `target`, `visibility?` |
| **Draw** | Взяти карт(и) з джерела. | `source`, `count`, `target?`, `distribution?`, `visibility?` |
| **Place** | Покласти карту з руки в партію (або іншу зону). | `source`, `target`, `visibility` |
| **Look** | Переглянути карт(и) (глядач бачить, інші ні). | `target_zone`, `count`, `visibility` |
| **Condition** | Здібність спрацьовує лише за умови. | `condition` |
| **Calculation** | Значення маркерів (або X) обчислюються з гри. | `x_source` |
| **PlayExtra** | Зіграти додаткову карту з обмеженнями. | `filters?` (наприклад `not_fraction`) |
| **Guess_Kill** | Вгадати карту й вбити її. | `target_player`, `target_zone`, `visibility` |
| **Kill_Random** | Вбити випадкову ціль у партії. | `target_player`, `target_zone`, `visibility` |
| **Kill_Dual** | Дві цілі одночасно (наприклад, своя та чужа face_down). | `targets`: масив зон |
| **Bury_Perform** | Поховати + виконати здатність (наприклад, з таверни). | `source` |
| **Perform** | Виконати здатність обраної карти (наприклад, у партії). | `target_zone`, `visibility` |
| **Perform_Top** | Виконати здатність верхньої карти (наприклад, з цвинтаря). | `source` |
| **Perform_Self** | Виконати здатність на своїй партії. | `target_zone`, `visibility` |
| **Swap_Hand** | Обміняти руки з іншим гравцем. | `target_player` |
| **Draw_All_Tavern** | Забрати всі карти з таверни з розподілом. | `distribution` |
| **Reveal_Harbor** | Відкрити N карт з гавані (з умовою). | `count`, `condition?` |
| **Move_Markers** | Змістити маркери на треку (leading/behind). | `options?` або `effects` + `logic?` |
| **Flip_Or_Look** | Вибрати: перевернути або переглянути. | `target_player`, `target_zone`, `visibility` |

### 3.3 Джерела карт (`source`)

Можливі значення (рядок або масив рядків):

- `"hand"` — рука активного гравця
- `"Tavern"` — таверна
- `"Harbor"` — гавань (колода)
- `["Harbor", "Tavern"]` — вибір джерела
- `"other_hand"` — рука супротивника
- `"face_up_party"` — відкриті карти в партії
- `"other_party"` — партія супротивника
- `"other_party_face_down"` — закриті карти в партії супротивника
- `"Graveyard"` — цвинтар (верх або кілька)
- `"Graveyard_top"` — тільки верхня карта цвинтаря
- `"self_party"`, `"self_party_face_down"`, `"self_hand"` — своя партія/рука

### 3.4 Цільові зони (`target`, `target_zone`, `targets`)

- `"Party"` — партія (відкриті/будь-які)
- `"Party_face_down"` — лише закриті в партії
- `"hand"` — рука
- `"Wilderness"` — пустош (сброс)
- Різні комбінації: `self_party`, `other_party`, `self_face_down`, `other_face_down` у полі `targets` (масив).

### 3.5 Видимість (`visibility`)

- `"face_up"` — лице вгору (відкриті)
- `"face_down"` — рубашкою вгору (закриті)

Використовується для фільтрації цілей (наприклад, Kill лише face_down карт).

### 3.6 Фільтри (`filters`)

- `fraction: "Undead"` — лише карти фракції Undead
- `fraction: "Highlanders"` — лише Highlanders
- У `PlayExtra`: `not_fraction: "Imperials"` — заборона грати карту фракції Imperials

Тип: `{ fraction?: string; not_fraction?: string }`.

### 3.7 Розподіл при Draw (`distribution`)

При кількох взятих картах — куди класти кожну:

- `{ "Party": 1, "Wilderness": 1 }` — одна в партію, одна в пустош
- `{ "Party_face_down": 1, "Wilderness": 1 }`
- `{ "Party_face_down": 1, "Wilderness": "rest" }` — одна в партію face_down, решта в пустош

Ключі: `Party`, `Party_face_down`, `Wilderness`. Значення: число або `"rest"`.

### 3.8 Умови спрацьовування (`condition`)

Унікальні значення з файлу:

| condition | Сенс (для імплементації) |
|-----------|---------------------------|
| `no_red_in_tavern` | У таверні немає червоних карт (Imperials). |
| `green_behind_red` | Зелений маркер позаду червоного на треку. |
| `has_blue_black_party` | У партії є сині/чорні (Waterfolk/Undead). |
| `has_red_party` | У партії є червоні (Imperials). |
| `red_behind_green` | Червоний позаду зеленого. |
| `has_face_down_undead` | Є закрита карта Undead. |
| `has_face_down_green` | Є закрита карта зеленої фракції. |
| `has_face_down_undead` | (дубль у таблиці — можна об’єднати з вище.) |
| `no_undead` | Немає Undead (наприклад, у гавані або на столі). |

Умови потрібно реалізувати на бекенді як предикати над станом гри (таверна, трек, партії, гавань).

### 3.9 Джерело для обчислення X (`x_source`)

Використовується в ability з `action: "Calculation"` і в маркерах з `"X"`/`"-X"`:

| x_source | Опис |
|----------|------|
| `target_party_markers` | Кількість маркерів у обраній партії (або сума). |
| `tavern_not_red` | Кількість карт у таверні не-червоної фракції. |
| `tavern_not_green` | Кількість не-зелених у таверні. |
| `target_face_up_green` | Кількість відкритих зелених у цілі. |
| `graveyard_count` | Кількість карт у цвинтарі. |
| `target_face_up_blue` | Відкриті сині (Waterfolk) у цілі. |
| `target_face_down_count` | Кількість закритих карт у цілі. |

На бекенді потрібно обчислювати ці величини зі стану і підставляти в формулу маркерів (red/green з "X"/"-X").

### 3.10 Move_Markers — особливості

Дві варіації:

1. **З `options` (масив рядків):**  
   `"options": ["-1 leading", "+2 behind"]` — гравець обирає один із варіантів (наприклад, -1 leading або +2 behind).

2. **З `effects` і `logic`:**  
   `"logic": "AND"` або `"OR"`, `"effects": ["-1 leading", "+1 behind"]` — застосувати обидва (AND) або один із двох (OR).

Підрядки типу `"-1 leading"`, `"+2 behind"` потрібно парсити: знак, число, ключове слово (leading/behind). **Leading** — маркер, що випереджає на треку; **behind** — той, що позаду.

## 4. Статистика по фракціях та ability

- **Highlanders:** 18 карт
- **Imperials:** 18 карт
- **Undead:** 18 карт
- **Waterfolk:** 18 карт  
Усього: **72 карти**.

Карти з `ability: null` — лише маркери, без додаткової здібності (наприклад, Depressed Druid, Short-sighted Soldier, Furious Pigman).

## 5. Рекомендації для бекенду

1. **Модель карти:** окрема сутність/таблиця або JSON-каталог з полями `name`, `fraction`/`faction`, `markers`, `ability`. Генерація `id` (наприклад `hero_1` … `hero_72`) при імпорті.
2. **Валідація:** схема (JSON Schema або Pydantic) для `markers` (включно з `logic`, `red_alt`, `green_alt`) і для кожної `action` з її полями.
3. **Обчислення маркерів:** окремий модуль для логік AND/OR/OR_NEG/OR_NEG_DECIDE_LEFT/AND_OR/LEADING_MARKER і для підстановки X (з `x_source` та стану гри).
4. **Обробники ability:** по одному handler на кожен `action` (Kill, Draw, Swap, Condition, Calculation, Move_Markers тощо), з перевіркою умов `condition` і фільтрів `filters`.
5. **Move_Markers:** чітке визначення правил «leading»/«behind» на треку (наприклад, за індексом клітинки) і парсинг рядків типу `"-1 leading"` / `"+2 behind"`.
6. **Ідемпотентність і черга ходів:** як і для поточних команд (PlayCard, DrawFromTavern тощо), визначити, в якій фазі й коли виконуються здібності (одразу при грі карти, чи окремим кроком з вибором цілей).

---

## 6. Рекомендації для фронтенду

1. **Відображення маркерів:** підтримка числових значень, "X", "-X" і альтернатив (red_alt, green_alt) з підказками (tooltip) про умови та X.
2. **Тексти здібностей:** локалізація (укр/ен) по `action` + параметрам (target_zone, source, visibility, condition, x_source); можливо шаблони на кшталт «Вбити закриту карту Undead у партії супротивника».
3. **UI для вибору:** для Draw — вибір джерела з `[Harbor, Tavern]`; для Kill/Bury/Flip — вибір цільової карти з фільтрами по visibility і fraction; для Move_Markers — вибір опції з `options` або `effects`; для Guess_Kill — введення/вибір назви карти.
4. **Condition:** показувати гравцю, чому здібність активна/неактивна (наприклад, «У таверні немає червоних»).
5. **Визначення leading/behind:** на треку візуально або текстом показувати, який маркер leading, щоб зрозуміло для Move_Markers.

## 7. Маппінг імен полів (поточний бекенд vs новий JSON)

| Новий JSON | Поточний бекенд (heroes) | Примітка |
|------------|--------------------------|----------|
| `fraction` | `faction` | Один до одного; при імпорті перейменувати або мапити. |
| `markers.red` / `markers.green` | `red_delta` / `green_delta` | Поточна модель — лише числа. Потрібно розширити на "X"/"-X" та логіку. |
| `markers.logic`, `red_alt`, `green_alt` | — | Додати в модель. |
| `ability` (цілий об’єкт) | `ability_id` (рядок) | Розширити до повної структури ability або зберігати ability_id + окремий каталог abilities. |

Після узгодження цього маппінгу можна згенерувати міграцію даних з нового `cards.json` у формат, який використовує поточний бекенд, або оновити бекенд під нову схему.
